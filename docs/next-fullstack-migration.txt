Essentialist — Next.js full-stack migration (incremental)

Are we moving `server/` (Express) into Next.js?
  No — not as a wholesale cutover. The plan is incremental:
  - **Today:** `server/` stays the canonical API, database access, and jobs.
  - **Next.js:** Owns the UI and can add Route Handlers that **proxy** (or later
    **re-implement**) Express routes after parity is verified.
  - **End state (optional, over time):** Some routes may live only in Next;
    others may stay on Express indefinitely. Nothing is removed from `server/`
    until you explicitly choose parity and traffic switching.

Git branch (workspace):
  feat/next-fullstack-migration

If you connect a remote after a fresh `git init` in this workspace:
  git remote add origin <url>
  git fetch
  (merge or rebase as needed — history may not match an older remote)

Canonical backend (unchanged):
  server/ — Express + Mongo + routes and jobs (see server/MIGRATION.txt)

Frontend — backend integration surface:
  client/src/backend/
    config/runtimePublicEnv.js     — env accessors
    config/upstreamOrigin.js       — getLegacyExpressOrigin() for server BFF fetches
    http/legacyClient.js             — re-export of src/utils/Axios.js
    http/axiosToastError.js          — re-export of src/utils/AxiosToastError.js
    contracts/summaryApi.js        — re-export of src/common/SummaryApi.js
    services/legacyExpressGateway.js — extension point for future gateways
    index.js                         — barrel exports

Path aliases (client/jsconfig.json):
  @/*        → client/src/*
  @backend/* → client/src/backend/*

Next.js Route Handlers (examples):
  client/src/app/api/system/migration-status/route.js — Next-only JSON probe
  client/src/app/api/legacy/health/route.js — read-only proxy to Express GET /health
    (env: API_URL, then NEXT_PUBLIC_API_URL)

Import migration status (app code):
  - Prefer: @/backend/http/legacyClient, @/backend/contracts/summaryApi,
    @/backend/http/axiosToastError (or imports from @/backend/index.js barrel).
  - Intentional exception: client/src/utils/Axios.js still imports
    ../common/SummaryApi directly for refresh-token wiring — avoids a circular
    graph if legacyClient re-exported Axios while Axios imported only from @backend.
  - utils/indexnow.js and hooks/useIndexNow.js use @/backend/contracts/summaryApi
    (replaces a broken ../api/SummaryApi path; useIndexNow now imports baseURL).

Live imports of @/utils/Axios, @/common/SummaryApi, or relative .../utils/Axios
outside backend/ and utils/Axios.js: none (only commented leftovers in a few files).

Conventions:
  - New code that calls Express should use @backend (or @/backend/...) paths.
  - Do not delete client/src/common/SummaryApi.js or client/src/utils/Axios.js
    until refresh logic is relocated and parity is tested.

Suggested next steps:
  1. Add domain Route Handlers under app/api/... that use `fetchLegacyExpress()`
     from `client/src/backend/services/legacyExpressGateway.js` to proxy Express;
     gate with tests or flags until responses match.
  2. Introduce request/response shapes under client/src/backend/contracts/types/
     (optional, incremental).
  3. Optionally add a monorepo workspaces root package.json for client + server.
  4. Remove stale commented import blocks in components/pages when touching those
     files for other reasons (low priority).
