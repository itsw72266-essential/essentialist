Essentialist — Next.js full-stack migration (incremental)

Are we moving `server/` (Express) into Next.js?
  You asked for a **complete full-stack Next.js app** without relying on an
  external Express process for those routes, while **`server/` stays intact**
  (reference + reuse for another app). The approach is **dual track**:
  - **`server/`** — unchanged; keep committing and running it as today.
  - **`client/src/fullstack/`** — new home for Zod schemas, Mongoose models,
    controllers, and Route Handlers that implement the same behavior against
    the **same `MONGODB_URI`** (and shared secrets like `RESEND_API`) when you
    wire traffic to `/api/next/...`.
  - **`client/src/backend/`** — legacy HTTP surface + optional **proxies** to
    Express (`/api/legacy/...`) during transition.

Conventions:
  - New code that calls Express should use @backend (or @/backend/...) paths.
  - Do not delete client/src/common/SummaryApi.js or client/src/utils/Axios.js
    until refresh logic is relocated and parity is tested.

Suggested next steps:
  1. Port more domains from `server/controllers` into `client/src/fullstack/controllers`
     and `client/src/app/api/next/...` (login, refresh, user-details, …), keeping
     JSON shapes compatible with the existing client.
  2. When a domain is ready, point the browser client at `/api/next/...` (or a
     `NEXT_PUBLIC_*` base URL flag) instead of Express for that call only.
  3. Add domain Route Handlers under app/api/legacy/... that use `fetchLegacyExpress()`
     if you still need Express during overlap.
  4. Introduce request/response shapes under client/src/backend/contracts/types/
     (optional, incremental).
  5. Optionally add a monorepo workspaces root package.json for client + server.
  6. Remove stale commented import blocks in components/pages when touching those
     files for other reasons (low priority).
